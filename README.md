## Knowledge Base — ChromaDB Store

Every agent created by KRE Agent Builder gets its own **isolated ChromaDB vector store** automatically provisioned during agent creation.

### Storage Location

```
backend/workspace/agents/<agent_id>/chroma_db/
├── chroma.sqlite3      # SQLite-backed vector index (cosine similarity, HNSW)
└── doc_meta.json       # Human-readable document metadata index
```

### What Gets Ingested Automatically

When the agent is saved or updated, the following documents are ingested into ChromaDB as part of the agent's personal knowledge base:

| Document | Description |
|----------|-------------|
| **Agent Identity** | Agent name, description, goal / purpose |
| **System Prompt** | The full LLM system prompt configured in the wizard |
| **Creation Prompt** | The original user prompt used to generate this agent |
| **Configuration** | LLM model, tools list, MCP servers, trigger settings, framework |
| **Training Conversation** | All Step-3 (system-prompt builder) chat messages |
| **Chat History** | The last 50 chat messages from previous sessions |
| **Generated Code** | First 2 000 characters of the agent's generated source code |
| **Inline Datasets** | JSON / CSV / Text / API datasets configured in the Data tab |

### Collection Settings

```python
chromadb.PersistentClient(path="<agent_id>/chroma_db")
collection = client.get_or_create_collection(
    name="knowledge_base",
    metadata={"hnsw:space": "cosine"},   # cosine similarity
)
```

> **Note:** Documents are stored as **text-only** (no explicit embedding vectors).  
> ChromaDB's built-in `sentence-transformers/all-MiniLM-L6-v2` (384-dim) handles  
> embedding at both upsert time and query time automatically, eliminating any  
> dimension-mismatch errors between different environments.

### Idempotent Ingestion

All document IDs are deterministic **MD5 hashes** derived from `agent_id + document_title`.  
Re-saving an agent simply **upserts** existing chunks — no duplicate data accumulates.

## Embedding Pipeline

```
Raw Text
   │
   ▼
_chunk_text()               # Split into 512-char chunks, 80-char overlap
   │
   ▼
collection.upsert(          # Store text-only in ChromaDB
    ids=[...],
    documents=[chunk_1, chunk_2, ...],
    metadatas=[{doc_id, title, source, chunk_index, ...}]
)
   │  (ChromaDB internally embeds each chunk with sentence-transformers, 384-dim)
   ▼
HNSW cosine index           # Persisted to chroma.sqlite3
```

At **query time** (during chat):

```
User message
   │
   ▼
collection.query(           # query_texts — ChromaDB embeds the query internally
    query_texts=[user_message],
    n_results=5
)
   │
   ▼
Top-5 chunks by cosine similarity
   │
   ▼
Injected into Gemini system prompt as a grounded context block
```

## Document Chunking

The chunking algorithm implemented in `rag_service/engine.py → _chunk_text()`:

| Parameter | Value | Description |
|-----------|-------|-------------|
| `CHUNK_SIZE` | **512 characters** | Maximum characters per chunk |
| `CHUNK_OVERLAP` | **80 characters** | Overlap between consecutive chunks (preserves context across boundaries) |
| Strategy | **Sliding window** | Moves forward by `CHUNK_SIZE - CHUNK_OVERLAP` each iteration |

### Example

```
Original text: 1 000 characters

Chunk 1: chars   0 – 511  (512 chars)
Chunk 2: chars 432 – 943  (512 chars, starts 80 chars before end of chunk 1)
Chunk 3: chars 864 – 999  (remaining characters)
```

Each chunk stored with metadata:
```json
{
  "doc_id": "abc123",
  "title": "My Agent — System Prompt",
  "source": "agent_profile",
  "chunk_index": 0,
  "total_chunks": 3,
  "ingested_at": "2025-04-14T12:00:00"
}
```

### Supported Input Formats

| Format | Conversion Strategy |
|--------|---------------------|
| **Plain text / Markdown** | Used directly |
| **JSON** | Recursively flattened to `key: value` lines |
| **CSV** | Converted to `header: value; header: value` rows |
| **PDF** | Text extracted page-by-page via `pypdf` |
| **DOCX** | Paragraph text extracted via `python-docx` |
| **URL / API** | Fetched, HTML stripped, truncated to 50 000 chars |

## RAG at Chat Time

Every message sent to `POST /chat` goes through this pipeline:

```
1. User sends message to POST /chat
2. rag_context_block(agent_id, user_message, top_k=5) is called
3. ChromaDB returns top-5 semantically similar chunks
4. Chunks are formatted as a context block:

   === KNOWLEDGE BASE CONTEXT ===
   [1] (Source: My Agent — System Prompt | Similarity: 0.923)
   <chunk text>

   ---

   [2] (Source: My Agent — Configuration | Similarity: 0.871)
   <chunk text>
   === END OF KNOWLEDGE BASE CONTEXT ===

5. Context block is prepended to the Gemini system prompt
6. Gemini generates a grounded response
7. Response returned to user
```
